### A1. `CMakeLists.txt:60` — `set(BUILD_TESTING OFF)` silently overrides `-DBUILD_TESTING=ON`

**Status:** VERIFIED

A plain (non-cache) directory-scope `set` placed before the multifloats fetch:

```cmake
if("multifloats" IN_LIST EPBLAS_PARALLEL_BUILT_TARGETS)
    set(BUILD_TESTING OFF)            # ← shadows cache for rest of this CMakeLists.txt
    set(MULTIFLOATS_BUILD_BENCH OFF)
    ...
endif()
```

The later `option(BUILD_TESTING "..." ON)` on line 224 won't recover: `option()` only sets the variable if it's *not already defined*, and the local var from line 60 is defined. When `if(BUILD_TESTING)` runs on line 225, the local OFF wins over whatever the cache says.

**Impact:** Whenever multifloats is in `EPBLAS_PARALLEL_BUILT_TARGETS`, tests silently don't build — even with `-DBUILD_TESTING=ON` on the command line. This is the underlying reason the project has been operating with `-DBUILD_TESTING=OFF` as a workaround (per `/tmp/handoff-pre-split-residue-20260528.md` "Out-of-scope" section).

**Fix:** scope the override. Either:
- Wrap the multifloats add_subdirectory call in a CMake function (function bodies have their own variable scope), or
- Use `set(BUILD_TESTING OFF CACHE BOOL "" FORCE)` and restore after `add_subdirectory`, or
- Move the `option(BUILD_TESTING ... ON)` ABOVE line 60 so user input is at least visible in cache before the override.

The function-scope approach is cleanest.

### A2. `cmake/epblas-parallelConfig.cmake.in:13` — OpenMP CXX dep not re-found by consumers

**Status:** VERIFIED

Config.cmake.in:
```cmake
find_dependency(OpenMP COMPONENTS C)
```

But `src/epblas-parallel/multifloats/CMakeLists.txt:128-129`:
```cmake
if(OpenMP_CXX_FOUND)
    target_link_libraries(${LIB_PREFIX}blas_parallel PUBLIC OpenMP::OpenMP_CXX)
elseif(OpenMP_C_FOUND)
    target_link_libraries(${LIB_PREFIX}blas_parallel PUBLIC OpenMP::OpenMP_C)
```

Result: the installed Targets file references `OpenMP::OpenMP_CXX` for the multifloats overlay (PUBLIC linkage, so it ends up in INTERFACE_LINK_LIBRARIES). The Config's `find_dependency(OpenMP COMPONENTS C)` does not create the `OpenMP::OpenMP_CXX` IMPORTED target on the consumer side. CMake will fail with "target OpenMP::OpenMP_CXX not found" at consumer-side `find_package(epblas-parallel)`.

**Impact:** Silent for the build-tree consumer (where the IMPORTED target was created at configure). Breaks any out-of-tree downstream that does `find_package(epblas-parallel)` after installation.

**Fix:** `find_dependency(OpenMP COMPONENTS C CXX)` in `cmake/epblas-parallelConfig.cmake.in:13`. Or generate the component list from `EPBLAS_PARALLEL_BUILT_TARGETS` (always include C; conditionally include CXX iff multifloats is built).

### B1. `tests/CMakeLists.txt:12` + `tests/epblas-parallel/CMakeLists.txt:50` — `fuzz_util.f90` OUTPUT collision

**Status:** VERIFIED

```cmake
# tests/CMakeLists.txt:10-24
function(add_fypp_source out_var shim_path body_dir)
    get_filename_component(_stem ${shim_path} NAME_WE)
    set(_out ${CMAKE_CURRENT_BINARY_DIR}/${_stem}.f90)   # ← no per-target segment
    ...
```

Called three times from `tests/epblas-parallel/CMakeLists.txt:50-52` (once per target in the `foreach(_target ...)` loop). The shim name is always `fuzz_util.fypp`, so all three iterations register an `add_custom_command(OUTPUT ${BUILD}/fuzz_util.f90)` — CMake keeps only one rule (iteration order is `kind10 kind16 multifloats`, so kind10's wins).

**Impact:** kind16 and multifloats `fuzz_common` static libs compile against the kind10-generated `.f90`. With `BUILD_TESTING=ON` (currently disabled by A1) this either fails at link time (kind/type mismatch) or — worse — silently runs the kind16/multifloats fuzz drivers against kind10 code, producing meaningless pass/fail results.

**Fix:** Add a per-target output subdir. Smallest patch:

```cmake
function(add_fypp_source out_var shim_path body_dir)
    get_filename_component(_stem ${shim_path} NAME_WE)
    get_filename_component(_target_dir ${shim_path} DIRECTORY)
    get_filename_component(_target_name ${_target_dir} NAME)   # "target_kind10" etc.
    set(_out ${CMAKE_CURRENT_BINARY_DIR}/${_target_name}/${_stem}.f90)
    file(MAKE_DIRECTORY ${CMAKE_CURRENT_BINARY_DIR}/${_target_name})
    ...
```

Or pass an explicit `target_subdir` arg — but the caller's shim path already carries it (`common/target_kind10/fuzz_util.fypp`).

### B2. `cmake/FortranCompiler.cmake:204-209` + `tests/epblas-parallel/CMakeLists.txt:54` — `fuzz_util.mod` race

**Status:** VERIFIED (related to B1; remains broken after B1 alone)

`fortran_module_layout` points every target at one shared dir:

```cmake
function(fortran_module_layout target)
  set(_moddir "${PROJECT_BINARY_DIR}/fmod/${FORTRAN_MOD_COMPAT_TAG}")
  set_target_properties(${target} PROPERTIES
    Fortran_MODULE_DIRECTORY "${_moddir}"
  )
  ...
```

The three per-target `*_parallel_fuzz_common` libs (and the openblas suite's `*_openblas_fuzz_common`) all define a module named `fuzz_util`. After B1 is fixed they all race to write `${_moddir}/fuzz_util.mod`.

**Impact:** Same as B1 (cross-target kind/type confusion or build break), surviving any B1-only fix.

**Fix:** per-target module dir for the fuzz_common libs:

```cmake
set_target_properties(${_fuzz_common} PROPERTIES
    Fortran_MODULE_DIRECTORY
    ${CMAKE_CURRENT_BINARY_DIR}/fmod/${_target})
target_include_directories(${_fuzz_common} PUBLIC
    ${CMAKE_CURRENT_BINARY_DIR}/fmod/${_target})
```

(Don't call the shared `fortran_module_layout` for these.)

### C1. `reports/cmp5/run_cmp5.sh:44` — `>&2 | tee -a "$LOG"` ordering bug

**Status:** VERIFIED

```bash
echo "[fatal] no ep_perf_* executables under $EP_DIR" >&2 | tee -a "$LOG"
exit 1
```

The `>&2` redirect runs before the pipe, so `tee` reads from empty stdin and writes nothing to `$LOG`. The fatal message reaches the user's terminal but not the log file.

**Fix:** `echo "..." | tee -a "$LOG" >&2`.

While there, also add a `[[ -d "$EP_DIR" ]] || { echo "[fatal] $EP_DIR missing" | tee -a "$LOG" >&2; exit 1; }` before the `find` so a missing dir is distinguishable from "no binaries".

### C2. `reports/cmp5/run_cmp5.sh:67-80` — partial `$TMP` on timeout flows into RAW.tsv

**Status:** PLAUSIBLE (verified the script flow; impact analysis from subagent)

```bash
if OMP_NUM_THREADS="$omp" ... timeout "$TIMEOUT" taskset -c 0 "$bin" > "$TMP" 2>>"$LOG"; then :
else
    echo "[fail] $run_id/$routine exit=$?" >> "$LOG"
fi
awk -v rid="$run_id" ... '...' "$TMP" >> "$RAW"
```

On `timeout` or crash, `$TMP` may contain header + a few data rows. The unconditional awk after the if/else ingests those partial rows. `aggregate.py` accepts them as measurements.

**Impact:** A routine that intermittently OOMs at large N produces "missing at large N" data — easy to mistake for "fast at small N, missing at large N". Wrong perf conclusions.

**Fix:** zero `$TMP` on failure, or skip the awk. Easiest:

```bash
if ! OMP_NUM_THREADS="$omp" ... timeout "$TIMEOUT" ...; then
    echo "[fail] ..." >> "$LOG"
    continue   # or: : > "$TMP"
fi
awk ... "$TMP" >> "$RAW"
```
